## 📚 核心概念

### 什么是 Planning with Files？
> 一个 Claude Code 技能系统，实现 Manus 风格的持久化 Markdown 规划——这是价值 20 亿美元收购背后的工作流模式。

### 核心价值
1. **持久化规划**: 规划保存在 Markdown 文件中，不是临时对话
2. **可追溯性**: 每个决策都有记录
3. **可迭代性**: 规划可以随时修改和更新
4. **可协作性**: 人类和 AI 可以共同编辑规划

---

## 🔧 工作流程

### 标准流程
```
1. 需求分析 → 创建 planning.md
2. 任务分解 → 更新 planning.md
3. 执行任务 → 标记进度
4. 遇到问题 → 更新规划
5. 完成任务 → 归档规划
```

### Planning.md 结构
```markdown
# 项目规划

## 目标
- [ ] 主要目标 1
- [ ] 主要目标 2

## 任务分解
### 阶段 1: 需求分析
- [ ] 任务 1.1
- [ ] 任务 1.2

### 阶段 2: 实现
- [ ] 任务 2.1
- [ ] 任务 2.2

## 进度追踪
- 开始日期：2026-03-08
- 预计完成：2026-03-15
- 实际完成：____

## 问题记录
### 问题 1
- 描述：...
- 解决方案：...
- 日期：...
```

## 💡 核心设计原则

### 1. 单一事实来源
- 所有规划集中在 planning.md
- 避免信息分散在对话历史中
- 新成员可以快速上手

### 2. 渐进式细化
- 从高层目标开始
- 逐步分解为具体任务
- 每个任务可执行、可验证

### 3. 显式状态追踪
- 每个任务有明确状态 (未开始/进行中/已完成)
- 进度可视化
- 阻塞问题及时标记

### 4. 问题驱动迭代
- 遇到问题立即记录
- 分析根本原因
- 更新规划避免重复

## 💡 关键洞察

### 1. 规划即代码
- 规划文件应该像代码一样版本控制
- 每次修改有提交记录
- 可以回滚到历史版本

### 2. AI 协作新模式
- AI 不是执行命令，而是参与规划
- 人类负责决策，AI 负责执行
- 规划是共同语言

### 3. 可追溯的价值
- 为什么做这个决策？
- 当时考虑了哪些因素？
- 如果重来会怎么做？

### 4. 迭代优于完美
- 规划不是一次性的
- 持续更新比完美规划更重要
- 灵活应对变化

## 📝 实施建议

### 开始步骤
1. **创建 planning.md**
   - 定义项目目标
   - 分解为可执行任务
   - 设置里程碑

2. **建立更新习惯**
   - 每天早上查看规划
   - 每天晚上更新进度
   - 遇到问题立即记录

3. **定期回顾**
   - 每周回顾进度
   - 调整下周计划
   - 总结经验教训

### 最佳实践
- ✅ 任务要具体、可衡量
- ✅ 状态要及时更新
- ✅ 问题要详细记录
- ✅ 决策要说明原因

### 避免的陷阱
- ❌ 规划过于详细（难以维护）
- ❌ 规划过于粗略（无法执行）
- ❌ 更新不及时（失去意义）
- ❌ 只规划不执行（纸上谈兵）
